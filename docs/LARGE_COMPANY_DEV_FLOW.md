# 大厂软件开发完整流程

> 以小米等一线互联网公司为例，一个桌面应用从想法到发布的完整流程。

## 流程总览

```
想法 → 市场调研 → 用户研究 → 立项评审 → 产品设计 → 技术方案 → 开发 → 测试 → 发布 → 运营
 │        │          │          │          │          │        │      │       │       │
 │     2-3周       2-3周      1-2周      4-6周      2-3周   8-16周  4-6周   1-2周   持续
 │                                                                        │
 └──────────────────── 总计：6-12个月 ──────────────────────────────────────┘
```

---

## 第一阶段：调研与立项（1-2个月）

### 1.1 想法验证（1-2周）

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 头脑风暴 | 团队讨论，提出想法 | 产品经理 + 团队 | 想法清单 |
| 竞品分析 | 分析市场上现有产品 | 产品经理 | 竞品分析报告 |
| 技术预研 | 评估技术可行性 | 技术负责人 | 技术可行性报告 |

### 1.2 用户调研（2-3周）

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 用户访谈 | 深度访谈目标用户 | 用户研究员 | 访谈记录 |
| 问卷调研 | 大范围收集需求 | 用户研究员 | 调研报告 |
| 数据分析 | 分析市场数据和趋势 | 数据分析师 | 数据报告 |
| 用户画像 | 建立目标用户模型 | 产品经理 | 用户画像文档 |

### 1.3 立项评审（1-2周）

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| PRD 撰写 | 产品需求文档 | 产品经理 | PRD 文档 |
| 预算评估 | 估算人力和资源成本 | 项目经理 | 预算表 |
| 立项答辩 | 向老板/委员会汇报 | 产品经理 + 技术负责人 | 立项文档 |
| 审批 | 领导审批 | 高层 | 立项批准 |

**关键决策点：**
- [ ] 市场需求是否真实存在？
- [ ] 技术方案是否可行？
- [ ] 投入产出比是否合理？
- [ ] 团队资源是否充足？

---

## 第二阶段：产品设计（1-2个月）

### 2.1 信息架构（1-2周）

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 功能拆解 | 确定所有功能模块 | 产品经理 | 功能清单 |
| 用户故事 | 描述每个用户场景 | 产品经理 | 用户故事列表 |
| 页面结构 | 确定页面层级和导航 | 产品经理 | 信息架构图 |

### 2.2 交互设计（2-3周）

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 线框图 | 画每个页面的低保真线框 | UI/UX 设计师 | 线框图（Figma） |
| 交互稿 | 定义每个按钮的行为、转场、异常 | UI/UX 设计师 | 交互稿 |
| 原型 | 做可点击的交互原型 | UI/UX 设计师 | 可交互原型 |

### 2.3 视觉设计（2-3周）

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 设计规范 | 定义颜色、字体、间距等规范 | 视觉设计师 | 设计规范文档 |
| UI 设计 | 设计每个页面的视觉稿 | 视觉设计师 | 设计稿（Figma） |
| 组件库 | 设计可复用的 UI 组件 | 视觉设计师 | 组件库 |
| 切图 | 导出开发需要的资源 | 视觉设计师 | 切图文件 |

### 2.4 技术方案（1-2周，与设计并行）

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 架构设计 | 确定整体技术架构 | 技术负责人 | 架构图 |
| 接口设计 | 定义前后端 API | 后端开发 | API 文档 |
| 数据库设计 | 设计数据模型 | 后端开发 | ER 图 |
| 技术评审 | 团队评审技术方案 | 全员 | 评审通过 |

---

## 第三阶段：开发（2-4个月）

### 3.1 开发准备（1周）

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 项目搭建 | 创建项目骨架、配置环境 | 前端/后端开发 | 可运行的空壳项目 |
| CI/CD | 配置持续集成和部署 | 运维 | 自动化流水线 |
| 代码规范 | 制定代码规范、ESLint 配置 | 全员 | 代码规范文档 |
| 分支策略 | 确定 Git 分支管理策略 | 技术负责人 | 分支策略文档 |

### 3.2 迭代开发（8-16周）

**敏捷开发节奏（每2周一个 Sprint）：**

| 环节 | 频率 | 做什么 | 谁做 |
|---|---|---|---|
| Sprint 计划会 | 每2周1次 | 确定本迭代要做的功能 | 全员 |
| 每日站会 | 每天1次（15分钟） | 同步进度和阻塞 | 全员 |
| 编码 | 按计划开发 | 开发工程师 | 前端/后端开发 |
| Code Review | 每个 PR | 审查代码质量 | 至少1人审查 |
| 单元测试 | 随代码编写 | 开发工程师 | 测试代码 |
| Sprint 评审 | 每2周1次 | 演示完成的功能 | 全员 |
| Sprint 回顾 | 每2周1次 | 反思改进点 | 全员 |

**一个 Sprint 的详细流程：**
```
Day 1:    Sprint 计划会 → 确定任务清单
Day 2-9:  每日站会 → 编码 → Code Review → 单元测试
Day 10:   Sprint 评审 → 演示 → Sprint 回顾
```

### 3.3 技术债务管理

| 类型 | 处理方式 |
|---|---|
| 临时方案 | 记录到技术债务清单，下个迭代处理 |
| 性能问题 | 评估影响，优先级高的立即修复 |
| 代码质量 | Code Review 中指出，合并前必须修改 |

---

## 第四阶段：测试（1-2个月）

### 4.1 内部测试（2-3周）

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 功能测试 | 逐个功能验证 | QA 工程师 | 测试报告 |
| 兼容性测试 | 不同系统/设备/分辨率 | QA 工程师 | 兼容性报告 |
| 性能测试 | 启动速度、内存、CPU | QA 工程师 | 性能报告 |
| 安全测试 | 漏洞扫描、权限验证 | 安全工程师 | 安全报告 |

### 4.2 Dogfooding（1-2周）

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 内部试用 | 全公司用这个产品 | 全员 | 反馈列表 |
| Bug 收集 | 汇总所有发现的问题 | QA 工程师 | Bug 清单 |
| 优先级排序 | 按严重程度排列 | 产品经理 | 修复优先级 |

### 4.3 Beta 测试（2-3周）

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 邀请用户 | 选择目标用户群体 | 产品经理 | Beta 用户列表 |
| 收集反馈 | 通过问卷/社区收集 | 用户运营 | 反馈报告 |
| 数据埋点 | 监控关键指标 | 数据工程师 | 数据面板 |

### 4.4 Bug 修复与打磨（2-4周）

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| P0 Bug | 严重 bug，必须修复 | 开发工程师 | 修复代码 |
| P1 Bug | 重要 bug，尽量修复 | 开发工程师 | 修复代码 |
| P2 Bug | 小问题，有时间再修 | 开发工程师 | Bug 清单（延后） |
| 性能优化 | 启动速度、渲染性能 | 开发工程师 | 优化代码 |

---

## 第五阶段：发布（1-2周）

### 5.1 发布准备

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 版本号 | 确定发布版本号 | 项目经理 | 版本号 |
| 发布文档 | 写更新日志和发布公告 | 产品经理 | 发布文档 |
| 官网更新 | 更新产品页面 | 设计 + 前端 | 官网 |
| 文档完善 | 用户指南、API 文档 | 技术文档 | 帮助文档 |

### 5.2 上架与发布

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 应用商店 | 提交到各平台商店 | 运维 | 上架通过 |
| 开源准备 | 整理开源仓库（如适用） | 开发工程师 | GitHub 仓库 |
| 营销素材 | 截图、视频、文案 | 市场团队 | 营销物料 |
| 发布日 | 正式发布 | 全员 | 产品上线 |

### 5.3 发布后

| 环节 | 做什么 | 谁做 | 产出 |
|---|---|---|---|
| 监控 | 监控崩溃率、性能 | 运维 | 监控面板 |
| 热修复 | 紧急 bug 修复 | 开发工程师 | 补丁版本 |
| 用户反馈 | 收集早期用户反馈 | 用户运营 | 反馈报告 |

---

## 第六阶段：运营与迭代（持续）

| 环节 | 做什么 | 频率 |
|---|---|---|
| 版本迭代 | 根据用户反馈持续改进 | 每2-4周 |
| 社区运营 | 维护用户社区、回复问题 | 每天 |
| 数据分析 | 分析使用数据，发现改进点 | 每周 |
| 技术分享 | 写博客、参加技术会议 | 每月 |

## 团队配置（典型）

| 角色 | 人数 | 职责 |
|---|---|---|
| 产品经理 | 1-2 | 需求定义、优先级管理、协调资源 |
| UI/UX 设计师 | 1-2 | 交互设计、视觉设计、组件库 |
| 前端开发 | 2-4 | UI 实现、交互开发 |
| 后端开发 | 2-3 | API 开发、数据存储、AI 集成 |
| QA 工程师 | 1-2 | 测试、质量保证 |
| 运维工程师 | 1 | 部署、监控、CI/CD |
| 项目经理 | 1 | 进度管理、风险管理 |
| **合计** | **9-15人** | |

## 与 LiveFront v2.0 的对比

| 环节 | 大厂做法 | LiveFront v2.0 做法 |
|---|---|---|
| 市场调研 | 2-3周，专人负责 | 你自己就是用户，省掉 |
| 用户研究 | 2-3周，访谈+问卷 | 你自己就是用户，省掉 |
| 立项评审 | 1-2周，向老板汇报 | 不需要，自己决定 |
| 产品设计 | 1-2月，设计师画稿 | 概念图 + 边做边设计 |
| 技术方案 | 2-3周，团队评审 | AI 辅助 + 自己判断 |
| 开发 | 2-4月，10人团队 | 2个月，1人+AI |
| 测试 | 1-2月，QA 团队 | 自己测 |
| 发布 | 1-2周，上架审核 | GitHub Release |
| 运营 | 持续，运营团队 | 自己维护 |

**核心差异：** 大厂用时间和人力换质量，你用 AI 换时间。
