# LiveFront v2.0 项目计划书

## 一、项目概述

### 1.1 项目背景
LiveFront v1.x 已发布，实现了文件树、Monaco 编辑器、实时预览、元素选中、属性面板、修改线、效果线、AI 对话、终端、Git 集成、MCP 双向互通等功能。但存在架构复杂、bug 频发、核心体验不聚焦的问题。

v2.0 将完全重写，以"前后端综合 AI 编辑器"为定位，让不会写代码的人也能从零搭建网站。

### 1.2 项目目标
用两个月时间，纯 AI 辅助开发，交付一个能让开发者自己用它做出自己网站的工具。

### 1.3 核心理念
> AI 是笔，LiveFront 是画板。

Vibe Coding 解决了"怎么让 AI 写代码"。LiveFront 解决了"AI 写完代码之后怎么办"。

### 1.4 资源与约束
| 项目 | 说明 |
|---|---|
| 团队 | 1人 + AI 辅助开发（Codex） |
| 预算 | 0 |
| 技术栈 | Electron + electron-vite + 纯 HTML/CSS/JS（无框架） |
| 时间 | 8周（2个月） |
| 目标用户 | 自己（验证概念可行性） |

## 二、市场调研

### 2.1 竞品分析

| 产品 | 定位 | 优势 | 劣势 | 与 LiveFront 的关系 |
|---|---|---|---|---|
| **Cursor** | AI 代码编辑器 | 强大的代码生成能力 | 仍是代码编辑器，视觉调整弱 | LiveFront 的上游——用户用 Cursor 写代码，在 LiveFront 调视觉 |
| **Bolt.new** | 浏览器端 AI 建站 | 零配置、即时预览 | 功能受限于 Web 端，无法操作本地文件 | 竞争者——但 LiveFront 是桌面端，文件操作更强 |
| **v0 (Vercel)** | AI 生成 UI 组件 | 组件质量高、React 生态 | 只生成组件，不做完整网站 | 补充者——LiveFront 可以调用 v0 生成的组件 |
| **Webflow** | 可视化建站 | 成熟的可视化编辑体验 | 不开源、付费、AI 能力弱 | 对标对象——LiveFront 的愿景是开源版 Webflow + AI |
| **Dreamweaver** | 传统网页编辑器 | 所见即所得 | 已过时、不支持现代框架 | 历史参考——LiveFront 是它的精神继承者 |

### 2.2 目标市场
Vibe Coding 用户群体：
- 不会写代码但想做网站的个人开发者
- 用 AI 写代码但在"调样式"上反复对话十次的开发者
- 想快速验证网页设计想法的设计师

### 2.3 差异化优势
1. **前后端一体化拓扑图**：唯一能同时可视化前端结构和后端关系的工具
2. **开源**：Webflow 等竞品是闭源付费的
3. **AI 原生**：不是"AI 辅助"，而是"AI 为核心"的编辑器
4. **素材库**：内置素材可拖拽，降低从零搭建的门槛

## 三、产品设计

### 3.1 信息架构

```
LiveFront v2.0
├── 文件管理
│   ├── 打开项目
│   ├── 文件树（极简，仅根文件）
│   └── 关闭项目
├── 预览区（核心）
│   ├── 设备切换（桌面/平板/手机）
│   ├── URL 栏
│   ├── 刷新/外部打开/分享
│   ├── 左键交互（正常点击）
│   └── 右键选中元素 → 属性面板
├── 右侧面板
│   ├── 属性面板（选中元素时）
│   │   ├── 标签/类名
│   │   ├── 排版（字号、行高）
│   │   ├── 颜色（背景、文字）
│   │   ├── 间距（上下左右）
│   │   ├── 边框
│   │   └── 效果（阴影、圆角等）
│   └── 终端（后端模式时）
│       ├── 运行后端服务
│       └── 查看运行结果
├── AI 对话区
│   ├── 对话输入
│   ├── 代码生成与应用
│   └── 项目文件操作（读取/修改/编写）
├── 底部工作流拓扑图
│   ├── 前端模块（方块）
│   ├── 后端关系（引导线）
│   ├── 交互关系（箭头）
│   ├── 点击定位
│   └── AI 自动分析
└── 素材库（隐藏在 Tab 中）
    ├── 背景素材
    ├── UI 组件
    ├── 动效预设
    └── 拖拽到预览区
```

### 3.2 核心交互流程

#### 流程一：从零搭建网站
```
1. 打开 LiveFront → 欢迎页
2. 点击"新建项目" → AI 询问需求
3. AI 生成项目文件 → 预览区显示白板
4. 用户从素材库拖拽背景/UI到白板上
   或直接告诉 AI "帮我做一个导航栏"
5. AI 修改代码 → 预览区实时更新
6. 底部拓扑图自动识别新增元素
7. 用户在拓扑图中定义前后端关系
   （如：登录按钮 → 调用后端登录接口）
8. 重复 4-7 直到网站完成
9. 一句话部署："帮我部署到服务器"
```

#### 流程二：编辑已有网站
```
1. 打开已有项目 → 预览区渲染网页
2. 右键点击元素 → 属性面板显示样式
3. 修改属性 → 预览区实时更新
4. 或告诉 AI "把按钮改成红色" → AI 自动改代码
5. 底部拓扑图显示完整结构
6. 点击拓扑图中的模块 → 定位到对应元素
7. 继续编辑 → 完成后导出/部署
```

### 3.3 界面布局

（参考概念图 docs/images/livefront-concept.png）

五个核心区域：
1. **文件树**（左侧）：极简，仅显示根文件
2. **预览区**（中间主体）：网页实时预览
3. **右侧面板**（右侧）：属性/终端双模式
4. **底部拓扑图**：前后端工作流可视化
5. **AI 对话区**（Tab 切换）：AI Agent

代码编辑区隐藏为高级选项。

## 四、技术方案

### 4.1 技术栈选择

| 组件 | 选择 | 理由 |
|---|---|---|
| 桌面框架 | Electron | 现有经验，跨平台，webview 原生支持 |
| 构建工具 | electron-vite | 快、配置简单 |
| UI 框架 | 纯 HTML/CSS/JS | 零依赖、启动快、适合 Electron |
| 代码编辑器 | Monaco Editor | 只在高级模式下使用，按需加载 |
| 终端 | node-pty + xterm.js | 成熟方案，v1.x 已验证 |
| AI 接口 | MiMo API（SSE 流式） | 核心差异化，v1.x 已集成 |
| 预览服务 | 内置 HTTP 静态服务 + 脚本注入 | v1.x 已验证 |

### 4.2 架构设计

```
┌─────────────────────────────────────────────────┐
│              Electron 主进程 (Node.js)            │
│                                                   │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐ │
│  │ 窗口管理  │ │ 文件系统  │ │ HTTP 预览服务    │ │
│  └──────────┘ └──────────┘ └──────────────────┘ │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐ │
│  │ AI API   │ │ 终端管理  │ │ 页面分析引擎     │ │
│  │ (MiMo)   │ │(node-pty)│ │ (DOM 解析)       │ │
│  └──────────┘ └──────────┘ └──────────────────┘ │
└──────────────────────┬──────────────────────────┘
                       │ IPC (ipcMain ↔ ipcRenderer)
┌──────────────────────┴──────────────────────────┐
│              Electron 渲染进程                     │
│                                                   │
│  ┌─── 核心层 ──────────────────────────────────┐ │
│  │ EventBus │ IPC Client │ 模块注册            │ │
│  └────────────────────────────────────────────┘ │
│                                                   │
│  ┌─── 功能模块 ───────────────────────────────┐ │
│  │ filetree │ preview │ props-panel           │ │
│  │ terminal │ ai      │ workflow-topology     │ │
│  │ materials│ settings                        │ │
│  └────────────────────────────────────────────┘ │
│                                                   │
│  ┌─── WebView 预览 ──────────────────────────┐ │
│  │  <webview> → HTTP Server                   │ │
│  │  注入：hover overlay + 元素选中 + postMessage│ │
│  └────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
```

### 4.3 模块清单（v2.0 精简版）

| 模块 | 职责 | 优先级 | 预估工时 |
|---|---|---|---|
| **filetree** | 极简文件树，仅显示根文件 | P0 | 1天 |
| **preview** | 实时预览 + 设备切换 + WebView 管理 | P0 | 2天 |
| **props-panel** | 元素选中 + CSS 属性可视化编辑 | P0 | 2天 |
| **ai** | AI 对话 + 文件操作（读/改/写） | P0 | 3天 |
| **terminal** | 内置终端 + 后端服务运行 | P1 | 1天 |
| **workflow-topology** | 页面拓扑图自动生成 | P1 | 5天 |
| **materials** | 素材库（背景/UI/动效） | P2 | 2天 |
| **settings** | 全局设置 | P2 | 1天 |

### 4.4 关键技术决策

**1. 元素选中方案**
- 复用 v1.x 的 Preview Server 脚本注入方案
- 在 HTML 响应中注入 hover overlay + click 选中脚本
- 通过 postMessage（非轮询）传递选中事件

**2. AI 文件操作**
- AI 对话时发送项目文件树 + 当前文件内容
- AI 返回结构化操作指令（修改文件/创建文件/删除文件）
- 主进程执行文件操作，渲染进程刷新预览

**3. 拓扑图生成**
- 两种模式：
  - **AI 分析模式**：将页面 HTML 发给 AI，AI 返回模块结构和交互关系
  - **用户标记模式**：用户手动点击元素，定义交互关系
- 拓扑图用 Canvas 或 SVG 渲染

**4. 编码问题**
- 所有源文件强制 UTF-8
- git 配置 .gitattributes 确保编码一致
- 读取文件时统一转码

## 六、测试方案

### 6.1 自测清单

每完成一个模块，开发者自己验证：

- [ ] 能启动应用，无崩溃
- [ ] 能打开文件夹
- [ ] 文件树正确显示
- [ ] WebView 能预览 HTML/CSS/JS
- [ ] 设备切换正常
- [ ] 能与 AI 对话
- [ ] AI 能修改文件
- [ ] 修改后预览区自动刷新
- [ ] 右键选中元素
- [ ] 属性面板显示正确
- [ ] 修改属性后预览区实时更新
- [ ] 拓扑图正确显示

### 6.2 边界情况

- 端口被占用时自动换端口
- 文件编码不是 UTF-8 时正确处理
- WebView 加载超时有提示
- AI API 调用失败时有错误处理
- 项目过大时文件树只显示根文件

## 七、开发流程规范

### 7.1 每日工作流程

**早上：**
1. 确认今天要完成的任务（从周计划中提取）
2. 启动 LiveFront v2.0（`npm run dev`），检查当前状态
3. 确认昨天的代码是否正常运行

**开发中：**
1. 每完成一个功能点，立即自测
2. 遇到问题先尝试自己解决，超过30分钟无进展则询问 AI
3. 每次修改后，用 `npm run dev` 启动验证，不打包
4. 避免在旧代码上反复修补，如果问题复杂，考虑重构

**下班前（或当天结束时）：**
1. 确保代码能正常运行，无崩溃
2. 提交代码：
   ```
   git add -A
   git commit -m "描述今天完成的工作"
   git push
   ```
3. 简单记录今天遇到的问题和明天的计划

---

### 7.2 Bug 修复流程

1. **发现问题**：记录 bug 现象（截图/文字描述）
2. **定位原因**：阅读相关代码，理解逻辑
3. **修复**：修改代码，从根源修复（检查文件编码、端口冲突等）
4. **验证**：用 `npm run dev` 启动，测试修复是否有效
5. **提交**：修复确认后提交推送

**关键原则：**
- 不要只改表面字符串，要从根源修复
- 端口冲突（9527/9528）要加错误处理，自动换端口或优雅失败，不要弹错误窗口
- 编码问题检查文件编码、移除 BOM，不只改表面字符串

---

### 7.3 新功能开发流程

1. **理解需求**：明确要做什么、为什么做、验收标准是什么
2. **设计**：思考实现方案，画简单流程图或写伪代码
3. **实现**：编写代码，遵循模块化架构
4. **自测**：功能测试 + 边界情况测试
5. **提交**：确认功能正常后提交推送

**模块开发规范：**
- 每个模块在 `src/renderer/modules/` 下创建独立目录
- 模块入口文件必须导出 manifest 对象（id/dependencies/commands/menus 等）
- 模块通过 `LiveFront.modules.register()` 注册
- 模块之间通过 EventBus 通信，避免直接引用

---

### 7.4 Git 提交规范

**提交格式：**
```
<type>: <description>
```

**Type 类型：**
| 类型 | 说明 | 示例 |
|---|---|---|
| feat | 新功能 | feat: 添加 AI 对话面板 |
| fix | 修复 bug | fix: 修复 WebView 预览白屏 |
| docs | 文档修改 | docs: 更新项目计划书 |
| style | 代码格式（不影响功能） | style: 统一缩进为 2 空格 |
| refactor | 重构（不增加功能/修复 bug） | refactor: 重构模块加载器 |
| test | 测试 | test: 添加预览模块测试 |
| chore | 构建/工具变动 | chore: 更新 electron-vite 配置 |

**示例：**
```
feat: 添加 AI 对话面板，支持 MiMo 模型流式输出
fix: 修复 Storage 持久化问题，重启后状态可恢复
docs: 添加项目计划书和开发流程规范
```

**分支管理（可选）：**
- `main` 分支：稳定版本，可发布
- `dev` 分支：开发版本，日常开发在 dev 分支进行
- 功能分支：`feature/xxx` 或 `fix/xxx`

---

### 7.5 打包发布流程

1. **确认所有功能正常**：跑一遍自测清单
2. **更新版本号**：修改 `package.json` 中的版本号
3. **打包**：执行 `npm run build`
4. **测试安装包**：安装并运行，确认无问题
5. **提交推送**：
   ```
   git add -A
   git commit -m "release: v2.0.0"
   git push
   ```
6. **创建 GitHub Release**：
   ```
   gh release create v2.0.0 --title "v2.0.0" --notes "..."
   ```

---

### 7.6 紧急修复流程

如果线上版本（已发布）发现严重 bug：
1. 创建 `hotfix/xxx` 分支
2. 修复 bug
3. 测试确认
4. 合并到 `main` 分支
5. 发布 patch 版本（如 v2.0.1）

---

### 7.7 代码质量规范

1. **命名规范**：
   - 变量/函数：camelCase（如 `openFile`, `fileTree`）
   - 常量：UPPER_SNAKE_CASE（如 `MAX_FILE_SIZE`）
   - CSS 类名：kebab-case（如 `file-tree`, `preview-container`）

2. **注释规范**：
   - 关键函数必须加注释说明用途
   - 复杂逻辑必须加注释解释思路
   - 不写无意义的注释（如 `// 设置变量`）

3. **错误处理**：
   - 所有异步操作必须有 try-catch
   - 文件操作必须有错误提示
   - 网络请求必须有超时处理

4. **性能规范**：
   - 避免在主线程做耗时操作
   - WebView 通信使用 postMessage，不用轮询
   - 文件监听要有防抖

## 八、风险与应对

| 风险 | 概率 | 影响 | 应对 |
|---|---|---|---|
| AI 文件操作不稳定 | 高 | 核心功能不可用 | 降级为"AI 建议 + 用户手动应用" |
| 拓扑图分析不准确 | 高 | 核心功能体验差 | 先支持用户手动标记，AI 辅助分析 |
| Electron 性能问题 | 中 | 启动慢、卡顿 | 精简模块、按需加载 |
| 两个月时间不够 | 中 | 功能不完整 | 优先保障 P0 模块，P1/P2 可延后 |
| 编码问题复发 | 低 | 乱码 | 从第一天起强制 UTF-8 + .gitattributes |
