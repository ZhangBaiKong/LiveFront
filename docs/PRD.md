## 一、产品概述

### 1.1 产品名称
LiveFront

### 1.2 一句话定位
AI 写完代码后的第一站——前后端综合 AI 编辑器。

### 1.3 Slogan
**AI 是笔，LiveFront 是画板。**

### 1.4 产品愿景
让不会写代码的人，也能在几分钟内做出一个专业网站。

### 1.5 目标用户
- **主要用户**：不会写代码但想做网站的个人（如 19 岁的开发者白空）
- **次要用户**：用 AI 写代码但在"调样式"上反复对话十次的开发者
- **潜在用户**：想快速验证网页设计想法的设计师

## 二、用户调研

### 2.1 用户画像

| 维度 | 信息 |
|---|---|
| 年龄 | 19 岁 |
| 背景 | 初中毕业，无代码基础 |
| 职业 | 无业在家，自学 AI |
| 编码能力 | 零（完全依赖 AI） |
| 使用场景 | 做个人网站、作品集、落地页、博客 |
| 目标 | 用 AI + 可视化工具从零搭建网站 |

### 2.2 用户痛点（来自用户调研）

**痛点一：看不懂代码**
- AI 生成一堆代码，用户看不懂
- 不知道哪些代码控制哪个元素
- 改代码就等于"把整个文件给 AI 重写"

**痛点二：流程太长**
- AI agent写完代码 → 打开浏览器 → 看效果 → 不满意 → 切回去 → 跟 AI 说 → 重复
- 每次改一个样式，都要走一遍完整流程

**痛点三：前端改了后端断了**
- 改了前端某个按钮的样式
- 这个按钮和后端接口的连接断开了
- 用户不知道怎么修

**痛点四：不知道什么是"好"**
- AI 给的设计可能很丑
- 用户不知道哪里丑、怎么改
- "看起来高级一点"这种描述，AI 理解不了

### 2.3 用户期望的理想体验

> 打开工具 → 告诉 AI "帮我做一个个人网站" → AI 自动从网上找优秀模板参考 → 短时间内生成网站 → 预览区直接看到效果 → 点一下元素就能改 → 改完直接发布

## 三、竞品分析结论

（引用 docs/COMPETITOR_ANALYSIS.md 的核心结论）

### 3.1 市场空白

| 能力 | Cursor | Bolt.new | v0 | Webflow | LiveFront |
|---|---|---|---|---|---|
| AI 代码生成 | 强 | 强 | 强 | 弱 | 强 |
| 可视化编辑 | 无 | 弱 | 弱 | 强 | **强** |
| 实时预览 | 有 | 有 | 有 | 有 | **有** |
| 元素选中修改 | 无 | 弱 | 无 | 强 | **强** |
| 前后端拓扑图 | 无 | 无 | 无 | 无 | **唯一** |
| 开源 | 否 | 否 | 否 | 否 | **是** |

**核心发现**：竞品要么只有 AI 没有可视化（Cursor/Bolt.new/v0），要么只有可视化没有 AI（Webflow），LiveFront 是唯一同时覆盖"AI + 可视化 + 拓扑图 + 开源"的产品。

### 3.2 竞品共同痛点
1. 改前端会断后端连接
2. AI 无法理解视觉审美
3. 没有全局结构视图
4. 无法拖拽调整
5. 前后端工作流割裂

### 3.3 LiveFront 的差异化壁垒
1. **前后端拓扑图**：唯一能同时可视化前端结构和后端关系的工具
2. **开源 + 本地**：数据不上传，可自配任何 AI
3. **素材库**：拖拽式搭建，降低从零开始的门槛

## 四、功能需求

### 4.1 功能模块（按优先级排列）

#### P0：核心最小可用产品（MVP）

| 模块 | 功能 | 用户故事 |
|---|---|---|
| **文件树** | 显示项目根文件 | 我能看到项目里有哪些文件 |
| **预览区** | 实时渲染网页 + 设备切换 | 我能直接看到网页效果 |
| **元素选中** | 右键选中元素，hover 高亮 | 我能点一下就知道这是什么元素 |
| **属性面板** | 修改元素的 CSS 属性（字号、颜色、间距等） | 我能直接改样式，不用写代码 |
| **AI 对话** | 内置 AI，能读写项目文件 | 我能告诉 AI 我想要什么，AI 直接改 |
| **AI 生成** | 从空白页面开始，AI 根据需求生成网站 | 我能一句话让 AI 做一个网站 |

#### P1：增强体验

| 模块 | 功能 | 用户故事 |
|---|---|---|
| **终端** | 运行后端服务 | 我能在软件里启动后端 |
| **设备预览** | 平板、手机尺寸切换 | 我能看不同设备上的效果 |
| **外部打开** | 在浏览器中打开当前页面 | 我能在真实浏览器中验证 |
| **导出** | 导出项目文件 | 我能把做好的网站下载到本地 |

#### P2：核心创新

| 模块 | 功能 | 用户故事 |
|---|---|---|
| **拓扑图** | 自动分析页面结构生成模块拓扑 | 我能看到整个网站的结构 |
| **前后端关系** | 可视化前端元素和后端接口的关系 | 我能知道每个按钮连着哪个后端接口 |
| **交互分析** | 自动识别点击跳转、弹窗等交互 | 我能看到哪些元素会触发页面跳转 |

#### P3：商业化功能

| 模块 | 功能 | 用户故事 |
|---|---|---|
| **素材库** | 背景、UI 组件、动效预设 | 我能拖拽素材到页面上 |
| **内置 AI 付费** | MiMo 特调模型，按 token 计费 | 我能用更懂设计的 AI |
| **模板市场** | 用户分享的网站模板 | 我能找到优秀模板快速开始 |
| **一键部署** | 部署到服务器 | 我能一句话发布网站 |

## 五、交互设计

### 5.1 界面布局

`
┌────────┬──────────────────────────────────┬──────────┐
│        │          预览区（主体）           │          │
│ 文件树 │                                  │ 属性面板 │
│ (左侧) │  ┌──────────────────────────┐    │ 或终端   │
│        │  │       网页实时预览       │    │ (右侧)   │
│        │  │                          │    │          │
│        │  └──────────────────────────┘    │          │
│        │  设备切换 | URL | 刷新 | 外部打开 │          │
├────────┴──────────────────────────────────┴──────────┤
│                   底部工作流拓扑图                      │
│    [Header] → [Hero] → [Cards] → [Footer] → [下载]   │
└──────────────────────────────────────────────────────┘
`

### 5.2 核心交互流程

**流程一：AI 从零生成网站**

1. 打开 LiveFront → 欢迎页
2. 点击"新建项目"
3. AI 对话区激活："你想做什么网站？"
4. 用户输入："帮我做一个个人作品集网站，暗色主题，有导航栏、Banner、作品展示区、关于我、联系表单"
5. AI 分析需求 → 从网络上参考优秀模板 → 生成代码文件
6. 预览区实时显示生成的网页
7. 底部拓扑图自动生成模块结构
8. 用户看到效果，觉得"导航栏颜色太浅"
9. 右键点击导航栏 → 属性面板显示 → 拖动滑块加深颜色
10. 或者告诉 AI："导航栏颜色再深一点"
11. 改完，预览区立刻更新
12. 重复直到满意 → 一键导出或部署

**流程二：编辑已有网站**

1. 打开已有项目文件夹
2. 预览区渲染网页
3. 右键选中元素 → 属性面板 → 修改样式
4. 底部拓扑图显示完整结构和前后端关系
5. 点击拓扑图中的"登录按钮"模块 → 跳转到对应元素
6. 发现"登录按钮 → 后端接口"的连接断了
7. 在拓扑图中重新连接 → AI 自动修复代码
8. 导出/部署

### 5.3 属性面板详细设计

选中元素时，右侧属性面板显示：

| 分类 | 属性 | 操作方式 |
|---|---|---|
| **信息** | 标签名（如 <button>） | 只读显示 |
| | 类名（如 .login-btn） | 可编辑文本 |
| **排版** | 字号 | 滑块 + 数值输入 |
| | 行高 | 滑块 + 数值输入 |
| | 字重 | 下拉选择（100-900） |
| | 对齐方式 | 按钮组（左/中/右） |
| **颜色** | 背景色 | 色板选择器 + 十六进制输入 |
| | 文字颜色 | 色板选择器 + 十六进制输入 |
| | 透明度 | 滑块 |
| **间距** | 内边距（上下左右） | 四向数值输入 |
| | 外边距（上下左右） | 四向数值输入 |
| **尺寸** | 宽度 | 数值输入 / auto |
| | 高度 | 数值输入 / auto |
| | 最大宽度 | 数值输入 |
| **边框** | 圆角 | 滑块 + 数值输入 |
| | 边框宽度 | 数值输入 |
| | 边框颜色 | 色板选择器 |
| **效果** | 阴影 | 预设选择 + 自定义 |
| | 过渡动画 | 预设选择 |

所有修改实时生效——用户改一个值，预览区立刻更新。

### 5.4 拓扑图详细设计

**自动分析规则：**
1. 扫描 HTML，识别所有可见元素（div、section、nav、footer、button 等）
2. 按视觉分组，合并为"模块"（如整个导航栏是一个模块，不是每个链接都是）
3. 识别交互：点击事件（onclick）、链接（href）、表单提交（submit）
4. 识别前后端关系：fetch/axios 调用、form action
5. 生成拓扑图：方块 = 模块，箭头 = 交互/跳转，虚线 = 后端连接

**展示格式：**

`
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ Header  │ ──→│  Hero   │ ──→│  Cards  │ ──→│ Footer  │
└─────────┘    │ [按钮]  │    └─────────┘    └─────────┘
               └────┬────┘
                    │
                    ↓
               ┌─────────┐
               │ Detail  │  ← 点击卡片跳转
               └────┬────┘
                    │
          ┌─────────┘
          ↓
    ┌─────────┐
    │/register│ ←─ ─ ─┘
    │ (后端)  │
    └─────────┘
`

## 七、技术需求

### 7.1 技术栈

| 组件 | 选择 | 理由 |
|---|---|---|
| 桌面框架 | Electron | 跨平台，webview 原生支持 |
| 构建工具 | electron-vite | 快，配置简单 |
| UI 框架 | 纯 HTML/CSS/JS | 零依赖，启动快 |
| 代码编辑器 | Monaco Editor（高级模式） | 按需加载 |
| 终端 | node-pty + xterm.js | 成熟方案 |
| AI 接口 | MiMo API（SSE 流式） | 核心差异化 |
| 预览服务 | 内置 HTTP 静态服务 + 脚本注入 | 已验证 |
| 拓扑图渲染 | Canvas 或 SVG | 待定 |

### 7.2 AI 能力需求

| 能力 | 说明 | 优先级 |
|---|---|---|
| 代码生成 | 从需求描述生成完整网站代码 | P0 |
| 代码修改 | 根据用户描述修改指定文件 | P0 |
| 页面分析 | 分析 HTML 结构，输出模块拓扑 | P1 |
| 交互识别 | 识别页面中的交互关系 | P1 |
| 前后端关系分析 | 识别前端元素和后端接口的连接 | P2 |
| 设计建议 | 根据设计原则建议配色、排版 | P2 |

## 九、风险与应对

| 风险 | 概率 | 应对 |
|---|---|---|
| AI 文件操作不稳定 | 高 | 降级为"AI 建议 + 用户手动应用" |
| 拓扑图分析不准 | 高 | 先支持用户手动标记 |
| 两个月时间不够 | 中 | 优先 P0，P2/P3 延后 |
| 做出来不好用 | 中 | 第 4 周就验证，及时调整方向 |
| 编码问题复发 | 低 | 从第一天起强制 UTF-8 |
